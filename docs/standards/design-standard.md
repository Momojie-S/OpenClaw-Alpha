# 设计文档规范

> 版本: v1.0
> 创建时间: 2026-03-03

---

## 核心原则

**设计文档 = 技术视角**，回答：
- 怎么实现
- 为什么这样设计
- 技术选型依据

---

## 边界判断

| 归属 | 判断标准 | 示例 |
|------|----------|------|
| **design** | 技术选型及原因 | 为什么用 SQLite 不用 MySQL |
| **design** | 架构设计 | 模块关系、数据流 |
| **design** | 接口契约 | 输入输出格式、数据结构 |
| **design** | 算法思路 | 关键算法的自然语言描述 |
| **code** | 具体实现 | 完整函数、语法细节 |

---

## 写作要求

### ✅ 应该写

- **架构/模块关系**：系统如何组织
- **技术选型及原因**：为什么选 A 不选 B
- **接口契约**：输入/输出/数据格式
- **数据模型**：表结构定义（不是 SQL 语法）
- **算法思路**：伪代码或自然语言描述
- **边界处理策略**：异常、边界如何处理

### ❌ 不应该写

- **完整函数实现**：代码在仓库里
- **大段代码块**：容易过时
- **语法细节**：变量名、语法糖
- **代码能自表达的**：让代码说话

### ⚠️ 边界情况

| 内容 | 建议 |
|------|------|
| 算法思路 | ✅ design（伪代码或自然语言） |
| 完整实现代码 | ❌ code |
| 接口定义（格式） | ✅ design |
| 接口调用示例 | ⚠️ 少量可以 |
| 数据表结构 | ✅ design（字段说明） |
| SQL 语句 | ❌ code |

## 数据源管理

**不在设计文档里列举 API 详细信息**，统一维护在 `docs/references/` 目录。

**设计文档只写关键信息**：
- 选用哪个接口（接口名）
- 关键字段（业务相关的几个）
- 特殊说明（如积分要求、缺失字段）

**完整信息引用 API 文档**：
```markdown
详见 [接口文档](../references/tushare/sw_daily.md)
```

**好处**：
- 避免每次网络搜索
- API 变更时只更新一处
- 设计文档保持精简

---

## 检查清单

在提交设计文档前，确认：

- [ ] 有技术选型及原因说明
- [ ] 接口契约清晰（输入/输出）
- [ ] 没有完整代码实现
- [ ] 算法用思路描述，不是代码
- [ ] 数据源引用统一文档
- [ ] 架构/数据流说明清楚
