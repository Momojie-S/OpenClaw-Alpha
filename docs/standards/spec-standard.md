# 需求文档规范

> 版本: v1.0
> 创建时间: 2026-03-03

---

## 核心原则

**需求文档 = 业务视角**，回答：
- 解决什么问题
- 为谁解决问题
- 如何验收

**技术无关性**：
- 换技术方案后，需求仍然成立
- 非技术人员能看懂
- 业务方可以确认和决策

---

## 边界判断

| 归属 | 判断标准 | 示例 |
|------|----------|------|
| **spec** | 业务方需要确认 | 权重配置、验收标准 |
| **spec** | 换技术方案仍有意义 | "计算热度指数" |
| **spec** | 业务领域知识 | "换手率 = 成交额/流通市值" |
| **spec** | 影响验收结果 | 验收标准 |
| **design** | 实现细节 | API、脚本、数据库 |
| **design** | 技术选型 | Python、SQLite、Tushare |
| **design** | 代码结构 | 目录结构、模块划分 |

---

## 写作要求

### ✅ 应该写

- **业务语言**：用户能理解
- **明确可测**：可以验证
- **业务规则**：公式、逻辑、权重
- **验收标准**：清晰的完成定义

### ❌ 不应该写

- **技术方案**：用什么 API、什么语言
- **代码实现**：函数名、变量名
- **表结构**：SQL、字段定义
- **具体工具**：Tushare、AKShare

### ⚠️ 边界情况

| 内容 | 建议 |
|------|------|
| 公式本身 | ✅ spec（业务规则） |
| 公式的代码实现 | ❌ design |
| 数据字段（业务含义） | ✅ spec |
| 数据字段（API 字段名） | ❌ design |
| 性能要求 | ✅ spec（如"3秒内返回"） |
| 如何实现性能 | ❌ design |

## 示例：产业热度追踪

### ✅ 正确的 spec 片段

```markdown
## 三、业务概念

### 3.1 热度指数

热度指数反映板块的市场关注度，由以下维度加权计算：

| 维度 | 权重 | 说明 |
|------|------|------|
| 涨跌幅 | 30% | 价格变动幅度 |
| 换手率 | 25% | 交易活跃程度 |
| 资金净流入 | 25% | 主力资金方向 |
| 涨跌家数比 | 20% | 板块内股票表现 |

### 3.2 趋势信号

| 信号 | 判断规则 |
|------|----------|
| 加热中 | 热度环比 > +20% 且 涨幅 > 0 且 资金流入 > 0 |
| 降温中 | 热度环比 < -20% 或 (涨幅 < -3% 且 资金流出) |
| 稳定 | 其他情况 |

## 四、验收标准

- [ ] 能输出板块热度排名（TOP 10）
- [ ] 热度指数计算符合上述公式
- [ ] 趋势信号判断符合上述规则
- [ ] 能对比上一交易日（环比）
```

### ❌ 错误的 spec 片段

```markdown

## 检查清单

- [ ] 非技术人员能看懂核心内容
- [ ] 没有具体的技术选型（API、工具、语言）
- [ ] 没有代码或表结构
- [ ] 业务规则明确（公式、权重、判断逻辑）
- [ ] 验收标准可测试
- [ ] 范围边界清晰（做/不做）

---

## 与设计文档的关系

| 文档 | 视角 | 内容 |
|------|------|------|
| spec.md | 业务 | 要什么、为什么、怎么验收 |
| design.md | 技术 | 怎么做、用什么、怎么实现 |

**协作流程**：spec → 业务方确认 → design → 基于 spec 的技术方案 → spec 变更时 design 同步更新
