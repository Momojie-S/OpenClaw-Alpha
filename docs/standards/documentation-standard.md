# 文档编写规范

## 核心原则

### 简洁优先
- 避免冗余，每句话都要有价值
- 用最少的文字传达核心信息
- 避免重复表述

### 设计导向
- 突出核心设计思路和架构决策
- 说明"是什么"和"为什么"
- 避免过度描述实现细节

### 无代码示例
- 文档中不包含具体实现代码
- 不包含 hello world 级别的示例
- 不粘贴配置文件或代码片段

### 无版本号和时间戳
- 文档中不记录版本号（如 v1.0, v1.1）
- 文档中不记录创建时间、更新时间
- 文档应该始终保持最新状态
- 通过 git 历史查看变更记录和创建时间

### 概念清晰
- 用自然语言描述设计思路
- 说明接口定义和数据流（如果需要）
- 避免让读者猜测设计意图

### 格式规范
- **列表行**: 列表项开始尽量不要使用加粗（`**`），仅在需要强调时使用
- **标题层级**: 合理使用标题层级，避免过多嵌套

## 文档应包含的内容

- **设计目标和核心思路**: 说明要解决什么问题，为什么这样设计
- **架构概览和模块关系**: 描述整体结构和各部分的关系
- **关键决策的背景原因**: 解释为什么选择某个方案，考虑了哪些因素
- **接口定义和数据流**: 如果需要，描述关键接口和数据流向
- **开发流程和规范要点**: 概括性的指导原则

## 文档不应包含的内容

- **具体实现代码**: 实现细节应通过代码本身和注释表达
- **代码示例**: 代码示例容易过时，增加维护负担
- **详细的配置文件内容**: 配置应通过配置文件本身和注释说明
- **IDE 使用教程**: 不作为项目文档的一部分
- **第三方库的基础教程**: 可通过链接引用官方文档

## 文档组织原则

### 按职责分类
- **specs/**: 需求文档（遵循 [spec-standard.md](./spec-standard.md)）
- **design/**: 设计文档（遵循 [design-standard.md](./design-standard.md)）
- **references/**: 数据源、接口清单等参考资料（遵循 [api-doc-standard.md](./api-doc-standard.md)）
- **architecture/**: 系统架构和技术选型
- **standards/**: 开发规范和文档规范

### Skill 文档结构

每个 skill 对应一个文档目录：

```
docs/industry-trend/
  spec.md              # 当前完整需求
  design.md            # 当前完整设计
  
  2026-03-03/          # 迭代 1（开发中可保留，完成后可删除）
    spec.md            # 本次迭代需求
    design.md          # 本次迭代设计
  
  2026-03-10/          # 迭代 2
    spec.md
    design.md
```

**迭代流程**：
1. 启动迭代 → 新建日期目录 `yyyy-MM-dd/`
2. 迭代期间 → 在日期目录下编写需求/设计
3. 迭代完成 → 合并更新主文档（spec.md / design.md）
4. 合并后 → 可删除日期目录（openspec/changes/ 有归档）

**与 OpenSpec 的关系**：
| 位置 | 内容 | 用途 |
|------|------|------|
| `docs/xxx/yyyy-MM-dd/` | 人可读的需求/设计 | 迭代过程 |
| `openspec/changes/xxx/` | spec-driven 开发流程 | 机器可读 + 归档 |

### 命名规范
- 使用 kebab-case 命名文件和目录
- 文件名应清晰反映内容主题
- 避免使用缩写和模糊的名称

## 文档维护

- 保持文档与代码同步更新
- 定期审查和优化过时内容
- 移除不再适用的文档
