# 策略回测功能研究报告

> 研究日期：2026-03-08
> 研究目标：评估在 OpenClaw-Alpha 中实现策略回测功能的可行性

---

## 一、背景

投资分析框架已完成基础能力建设，P2 高级功能中的**策略回测**是验证投资策略有效性的关键工具。

**价值**：
- 验证投资策略的历史表现
- 评估策略风险收益
- 优化策略参数
- 增强投资决策信心

---

## 二、技术调研

### 2.1 主流回测框架

| 框架 | 特点 | 优势 | 劣势 | 适用性 |
|------|------|------|------|--------|
| **Backtrader** | 灵活、模块化 | Pythonic、支持实盘、功能丰富 | 需要数据管理 | ⭐⭐⭐⭐⭐ |
| **Zipline** | 事件驱动 | 学术研究、Pipeline API | 维护放缓、安装复杂 | ⭐⭐⭐ |
| **VectorBT** | 向量化 | 极速回测 | 不适合复杂策略 | ⭐⭐⭐⭐ |
| **QuantConnect** | 云端 | 数据丰富 | 依赖云服务 | ⭐⭐ |

**推荐选择**：**Backtrader**

**理由**：
1. 灵活性高，支持多种策略类型
2. 支持 PandasData，可直接使用我们的数据
3. 内置丰富的技术指标和分析器
4. 支持实盘交易（未来扩展）
5. 活跃的社区和完善的文档

### 2.2 数据源支持

| 数据源 | 回测适用性 | 我们的现状 | 备注 |
|--------|-----------|-----------|------|
| **Tushare** | ⭐⭐⭐⭐⭐ | ✅ 已支持 | 历史数据完整、稳定 |
| **AKShare** | ⭐⭐⭐⭐ | ✅ 已支持 | 免费、数据丰富 |

**优势**：
- 我们已经有完整的历史数据获取能力
- 无需额外的数据源接入
- Fetcher 框架可直接复用

### 2.3 A股特殊考虑

| 问题 | 说明 | 处理方案 |
|------|------|----------|
| **涨跌停板** | 市价单可能失真 | 自定义 Broker 逻辑 |
| **交易费用** | 佣金、印花税、过户费 | 准确设置手续费 |
| **除权除息** | 股价调整 | 使用前复权/后复权数据 |
| **停牌** | 无法交易 | 数据清洗时处理 |
| **T+1 交易** | 当日买入次日才能卖 | 策略中限制 |

## 三、架构设计

### 3.1 模块关系

```
backtest/
├── backtest_processor.py          # 回测主流程
├── data_adapter.py                # 数据转换（Fetcher → Backtrader）
├── strategies/                    # 策略库
│   ├── base_strategy.py           # 策略基类
│   ├── ma_cross_strategy.py       # 均线交叉策略
│   └── rotation_strategy.py       # 行业轮动策略
└── analyzers/                     # 分析器扩展
    └── performance_report.py      # 性能报告
```

### 3.2 数据流

```
1. 用户输入：股票列表、策略、日期范围
    ↓
2. Fetcher 获取历史数据
    ↓
3. DataAdapter 转换为 Backtrader 格式
    ↓
4. BacktestProcessor 运行回测
    ↓
5. 输出结果：
   - 控制台：关键指标（收益率、夏普比率、最大回撤）
   - 文件：完整交易记录、资金曲线
```

### 3.3 与现有框架的集成

| 组件 | 复用情况 | 说明 |
|------|---------|------|
| **Fetcher** | ✅ 直接复用 | 历史数据获取 |
| **Processor 模式** | ✅ 直接复用 | 命令行入口、双输出 |
| **数据存储** | ✅ 直接复用 | .openclaw_alpha/ 目录 |

## 四、实现方案

### 4.1 MVP 范围

**核心功能**：
1. 单股票回测
2. 2-3 种基础策略（均线交叉、行业轮动）
3. 关键指标输出（收益率、夏普、回撤）

**不包含**：
- 多股票组合回测
- 参数优化
- 实时策略推送
- 可视化图表

### 4.2 技术栈

| 组件 | 选择 | 版本要求 |
|------|------|---------|
| 回测引擎 | backtrader | >=1.9.76 |
| 数据处理 | pandas | 已有 |
| 数据源 | Tushare/AKShare | 已有 |

### 4.3 开发计划

| 阶段 | 任务 | 预计时间 |
|------|------|---------|
| Phase 1 | 数据转换模块 | 2 小时 |
| Phase 2 | 策略基类 + 均线策略 | 2 小时 |
| Phase 3 | 回测主流程 | 2 小时 |
| Phase 4 | 单元测试 | 2 小时 |
| Phase 5 | 文档编写 | 1 小时 |
| **总计** | | **9 小时** |

## 五、风险评估

### 5.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| A股涨跌停处理复杂 | 中 | 高 | MVP 先不做特殊处理，后续优化 |
| 历史数据量大 | 高 | 中 | 限制回测时间范围（默认 1 年） |
| Backtrader 学习曲线 | 低 | 低 | 参考官方文档和社区案例 |

### 5.2 业务风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 策略过拟合 | 高 | 高 | 警告用户，强调历史表现不代表未来 |
| 数据准确性 | 低 | 高 | 使用 Tushare 等可靠数据源 |

## 六、结论

### 6.1 可行性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术可行性 | ⭐⭐⭐⭐⭐ | Backtrader 成熟，数据源就绪 |
| 开发成本 | ⭐⭐⭐⭐ | 约 9 小时，可控 |
| 业务价值 | ⭐⭐⭐⭐⭐ | 提升分析框架完整度 |
| 优先级 | ⭐⭐⭐ | P2 高级功能，非紧急 |

**总评**：✅ **可行**

### 6.2 建议

1. **优先级**：P2 中等优先级，可在完成预警推送后开发
2. **MVP 范围**：先实现单股票、基础策略，验证架构
3. **迭代方向**：
   - V1：单股票 + 基础策略
   - V2：多股票组合
   - V3：参数优化 + 可视化
