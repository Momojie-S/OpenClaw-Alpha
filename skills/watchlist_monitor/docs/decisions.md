# 决策记录 - watchlist_monitor

## 调研

### 性能瓶颈分析

**问题**：
- 原实现使用 `stock_zh_a_spot_em()` 获取全市场行情（5000+ 只股票）
- AKShare 内部进行 58 次分批请求，每次约 1 秒，总共约 58 秒
- 即使只需要 5 只自选股，也要获取全市场数据

**AKShare 限制**：
- 全市场行情接口的设计限制，无法通过代码优化解决
- 这是数据源本身的性能瓶颈

### 个股行情接口调研

**接口**：AKShare `stock_individual_spot_xq()`
- 优点：单次请求，响应快（约 0.5 秒/只）
- 缺点：需要雪球格式的股票代码（如 "SH600000"）
- 适用场景：自选股数量较少（<= 50）

**性能对比**：
| 场景 | 全市场接口 | 个股接口 | 提升 |
|------|-----------|---------|------|
| 5 只自选股 | 58 秒 | 2.8 秒 | 20x |
| 10 只自选股 | 58 秒 | ~5.6 秒 | 10x |
| 50 只自选股 | 58 秒 | ~28 秒 | 2x |

## 决策

### 优化方案：根据数量选择数据源

**策略**：
- 自选股数量 <= 50：使用个股行情 API（`stock_individual_spot_xq()`）
- 自选股数量 > 50：使用全市场行情 API（`stock_zh_a_spot_em()`）

**阈值选择**：
- 50 只是权衡点：
  - < 50 只：个股接口更快
  - > 50 只：全市场接口开始有优势（虽然仍需 58 秒）

### 实现设计

**新增 Fetcher**：
- `StockIndividualSpotFetcher`：使用个股行情 API
- 支持雪球格式代码转换

**代码转换**：
- 6 位代码（"000001"）→ 雪球格式（"SZ000001"）
- 规则：6 开头加 "SH"，其他加 "SZ"

**Processor 修改**：
- 在 `watch()` 方法中根据自选股数量选择获取方式
- 保持接口不变，对用户透明

## 踩坑

### 雪球代码格式

**问题**：
- AKShare 个股接口需要雪球格式代码
- 雪球格式：沪市前缀 "SH"，深市前缀 "SZ"

**解决**：
- 创建代码转换工具函数
- 在 Fetcher 内部自动转换

### 并发请求

**问题**：
- 个股接口需要逐个请求
- 串行请求 50 只股票仍需约 28 秒

**考虑**：
- 可使用异步并发请求优化
- 但需注意 API 限流风险
- 当前暂不实现，留给后续优化

## 优化经验总结（2026-03-11）

### 性能优化流程

1. **定位瓶颈**：
   - 使用日志和计时定位耗时操作
   - 分析是代码问题还是外部依赖问题

2. **调研替代方案**：
   - 寻找更高效的 API（个股 vs 全市场）
   - 评估性能提升空间

3. **设计权衡**：
   - 根据数据量选择合适的方案
   - 设置合理的阈值（如 50 只）

4. **保持接口稳定**：
   - 内部优化，外部接口不变
   - 对用户透明

### 通用经验

**数据源选择策略**：
- 数据量少：使用精确查询 API
- 数据量多：使用批量查询 API
- 设置阈值动态选择

**代码转换处理**：
- 不同 API 的代码格式可能不同
- 在 Fetcher 内部统一处理转换
- 保持内部使用统一格式（如 6 位代码）

**性能优化边界**：
- 区分代码优化和数据源限制
- 数据源限制无法通过代码优化解决
- 考虑缓存、预加载等方案
