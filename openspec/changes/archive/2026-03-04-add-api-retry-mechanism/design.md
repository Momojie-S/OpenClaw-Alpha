## Context

DataFetcher 实现规范文档中已描述了自动重试机制的设计，但代码层面尚未实现。当前所有 Fetcher 的 API 请求方法都缺乏重试能力，遇到临时性故障时会直接失败。

约束：
- 使用 tenacity 库作为重试框架（Python 生态主流选择）
- 重试机制只应用于 API 请求方法，不用于数据转换方法
- 需要区分可重试异常和不可重试异常

## Goals / Non-Goals

**Goals:**
- 为 API 请求方法提供统一的自动重试能力
- 定义清晰的异常分类，确保只有临时性故障才触发重试
- 使用指数退避策略避免雪崩效应
- 最小化对现有代码的改动

**Non-Goals:**
- 不为数据转换方法添加重试（转换逻辑应是纯函数）
- 不提供自定义重试配置接口（统一使用默认策略）
- 不测试重试机制本身（由 tenacity 库保证）

## Decisions

### 使用 tenacity 库

**选择 tenacity 的原因：**
- Python 生态中最成熟的重试库
- 支持指数退避、条件重试等高级特性
- 装饰器模式，侵入性小

**替代方案：**
- 手写重试逻辑：代码冗余，容易出错
- 其他库（retrying、backoff）：不如 tenacity 活跃

### 异常分类策略

**可重试异常：**
- RateLimitError（429）：限流是临时的
- TimeoutError：网络超时可能恢复
- ServerError（5xx）：服务端临时故障
- NetworkError：网络连接问题

**不可重试异常：**
- AuthenticationError（401）：认证失败不会因重试解决
- PermissionError（403）：权限问题需要人工处理
- NotFoundError（404）：资源不存在
- ValidationError（400）：参数错误

### 重试策略参数

| 参数 | 值 | 理由 |
|------|---|------|
| 最大重试次数 | 3 | 平衡可靠性和响应时间 |
| 等待策略 | 指数退避 | 避免雪崩，1s → 2s → 4s |
| 最大等待时间 | 30s | 避免无限等待 |

## Risks / Trade-offs

**重试可能延长响应时间** → 对于实时性要求高的场景，3 次重试最长可能等待 7 秒。这是可接受的代价，因为失败比慢更糟糕。

**异常转换遗漏** → 某些第三方库的异常可能未被正确分类。需要在实现时仔细处理各数据源 SDK 的异常类型。
